# CONTAINER 18-Month Startup Roadmap v2

Date: 2026-06-15
Status: Revised startup roadmap for v2 architecture

## Summary

CONTAINER should develop from an early lunar construction concept into a credible startup-grade technical program. The v2 roadmap keeps the original evidence-first startup strategy but resets the technical plan around the open-top container, front sliding-lift bucket, four side-rail robotic arms, and no-anchor stability challenge.

Default strategy: learn fast, prove the risky assumptions cheaply, and pitch only what has evidence behind it.

## Startup Positioning

CONTAINER v2 is an open-top lunar regolith construction machine. It is designed to land lighter, collect local regolith into its body, use carried regolith as working mass, and support landing pad and berm construction with a front bucket and four side-rail robotic arms.

Do not position CONTAINER v2 as:

- A complete lunar construction solution.
- A final or validated design.
- Ready for integration.
- Cheaper than alternatives without a sourced cost model.
- Proven stable without no-anchor modeling and physical tests.
- Independent of power, dust, thermal, mobility, arm, bucket, or autonomy risk.

## Phase 1: v2 Baseline Control, Months 0-2

Objective: make v2 the technical source of truth and prevent old v1 claims from leaking into public materials.

Actions:

- Use `../00_Current_Baseline/CONTAINER_Baseline_v2.md` as the current technical truth.
- Keep v1 documents, but label them as historical or superseded when referenced.
- Maintain an evidence confidence system: calculated, project assumption, external source, test result, design target, or retired.
- Replace final-design wording with: "revised baseline pending validation."
- Build the first advisor list across lunar systems, robotics, regolith/geotechnical, power/thermal, aerospace business, and startup/IP.

Deliverables:

- v2 source-of-truth index.
- v2 founder brief.
- v2 top technical unknowns.
- v2 experiment roadmap.
- Advisor outreach list and email templates.

Success criteria:

- Every important number has an owner, confidence level, and next verification method.
- Public-facing materials do not describe gantry, anchors, outriggers, or old intake as current v2 systems.
- The project can be explained honestly in less than 2 minutes.

## Phase 2: v2 Technical Credibility Sprint, Months 2-5

Objective: make the revised architecture hard to dismiss by improving the model around the assumptions most likely to break v2.

Actions:

- Build a v2 model covering volume, mass, center of mass, stability, bucket loads, arm workspace, mobility, power, thermal, schedule, and cost ranges.
- Add kill-case scenarios for no-anchor stability, bucket loads, arm stiffness, dust durability, battery survival, mobility/ground pressure, and cost.
- Convert model results into charts suitable for advisor review.
- Update the proof-versus-assumption table after each model revision.

Deliverables:

- v2 model update.
- Sensitivity charts.
- Risk-ranked v2 experiment roadmap.
- Revised technical brief.
- Current proof-versus-assumption table.

Success criteria:

- The model identifies the top feasibility drivers.
- Weak assumptions are visible instead of hidden.
- Advisor questions can be answered with current evidence or marked as open.

## Phase 3: Low-Cost v2 Proof Experiments, Months 5-10

Objective: run subscale experiments before attempting an integrated machine.

Actions:

- Test bucket excavation, lift, dump, and jam behavior in simulant.
- Test open-container fill, dump, bridging, and center-of-mass behavior.
- Test a side-rail arm/carriage module for reach, payload, stiffness, and dust tolerance.
- Test arm docking/coupler alignment, load transfer, and release.
- Test dust exposure effects on rails, bucket mechanisms, arm joints, sensors, radiator surfaces, and rear hatch seals.
- Test sintering coupon or alternative pad-surface construction behavior.

Deliverables:

- Test reports with photos, videos, and measured values.
- Measured values for the assumptions register.
- Prototype failure log.
- Revised architecture decision: continue, pivot, narrow scope, or retire a subsystem approach.

Success criteria:

- Each test produces measured values, not only visual proof.
- Failed tests update the design.
- Public claims are revised to match evidence.

## Phase 4: Startup Formation Package, Months 8-12

Objective: turn the technical work into a package suitable for advisors, competitions, incubators, grants, space accelerators, and early investor conversations.

Actions:

- Define the first beachhead as repeatable lunar landing pad and berm support using locally collected regolith.
- Prepare the pitch deck, one-page investor/advisor memo, technical appendix, and demo budget.
- Start a basic IP and prior-art review.
- Document which information can be shared publicly and which should remain confidential.

Deliverables:

- 10-12 slide pitch deck.
- One-page investor/advisor memo.
- Technical appendix.
- IP/prior-art notes.
- Budget for next demonstrator.
- Advisor feedback summary.

Success criteria:

- The startup story is specific, technical, and evidence-first.
- The next funding milestone has a clear cost, output, and reason.
- The project can survive basic diligence questions without overclaiming.

## Phase 5: Integrated Demonstrator and Funding Path, Months 12-18

Objective: build a visible integrated demonstrator only after individual subsystem tests produce useful data.

Actions:

- Build a bench or yard-scale demonstrator showing bucket excavation, fill, center-of-mass management, arm handling, dump/place behavior, and relocation logic.
- Measure cycle time, power draw, fill/dump behavior, bucket load, arm repeatability, dust sensitivity, and failure modes.
- Use the demo to pursue grants, accelerator entry, a university lab partnership, or early seed conversations.

Deliverables:

- Integrated demo video.
- Measured cycle data.
- Updated cost and mass model.
- Seed/grant proposal package.
- 24-month technical development plan.

Success criteria:

- The demo shows the v2 operating cycle at subscale.
- The measured data improves the model.
- The next raise or partnership ask is tied to a specific technical milestone.

## Monthly Cadence

Each month should end with:

- One updated learning note.
- One updated risk or assumption entry.
- One advisor question or outreach action.
- One model, test, CAD, or document artifact.
- One decision about what to stop, continue, or narrow.
